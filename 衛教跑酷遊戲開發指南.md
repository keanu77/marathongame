# 衛教網頁跑酷遊戲開發指南（整合版）

本指南整合「馬拉松完賽訓練」的實際開發經驗，以及後續玩法、程式、衛教與部署審查所得的建議，提供給想製作**手機優先、短局制、具有健康教育目的的 2D 網頁遊戲**之開發者與內容設計者參考。

這類遊戲適合一般健康教育與互動推廣，不應用來進行醫療診斷、風險預測或個人化治療與訓練處方。

## 如何閱讀本指南

文中的建議分成三種層級：

- **通用原則**：跨題材、遊戲引擎與雲端平台仍大致成立。
- **建議起始值**：可作為原型預設，但必須經模擬、真機與真人測試調整。
- **本專案經驗**：來自「馬拉松完賽訓練」、Phaser 3、TypeScript 與 Cloudflare Pages／D1 的具體做法，不代表唯一解法。

## 一、先定義成功：可玩性 × 學習 × 安全

衛教遊戲不能只看「好不好玩」，也不能反過來認為好玩不重要。更精確的原則是：

> **好玩與容易理解，是知識能被觸達的前提；正確學習與安全，才是最終目的。**

開始寫程式前，先回答：

1. 目標玩家是誰？他們原本知道多少？
2. 玩家多久能理解操作與目標？
3. 一局結束後，希望玩家帶走哪一至三個觀念？
4. 哪些誤解是絕對不能由玩法暗示的？
5. 玩家中途停止時，是否仍能得到有價值的內容？

建議同時觀察五類指標：

- **玩法理解**：玩家是否知道何時跳躍、收集與避開？
- **完成漏斗**：各關抵達率、停止原因、平均遊戲時間。
- **知識觸達**：每則衛教實際被多少玩家看見？
- **學習與誤解**：玩家是否能用自己的話說出重點？有沒有學到相反觀念？
- **體驗意願**：是否願意再玩一次、分享或推薦？

### 建議起始值

- 單局可先設計為約 60～90 秒。
- 操作說明應盡量在 30～60 秒內被理解。
- 一局帶走一至三個核心觀念，結算回顧最多約三至五點。

以上是短局衛教遊戲的原型起點，不是通用標準。

## 二、機制就是訊息：玩法必須與衛教一致

玩家從獎勵、損失與選擇中學到的，通常比文字更深。因此每個機制都要問：

> 玩家反覆玩過後，會把什麼行為理解成「好」或「壞」？

### 避免純陷阱道具

如果「間歇訓練」只會讓畫面更快、耗能更多，卻沒有任何收益，玩家會自然學到「間歇訓練是壞事」。較合理的做法是讓它有真實的成本與回報，例如：

- 增加速度與耗能，同時提前下一次補給機會；
- 在可驗證的條件下增加分數或資源；
- 只有在體力充足時才容易發揮效益。

重點不是一定要「雙倍分數」，而是設計文件所說的取捨，在遊戲規則中真的存在。

### 健康完成優先於硬撐

計分與排行榜可以傳達：

- 完賽者優先於中途停止者；
- 同分有清楚且公平的並列規則；
- 保留體力、降低遊戲風險可形成「安全完成」的正向回饋。

若加入「健康完賽評級」，必須標明它只是遊戲評級，不是醫療風險或身體狀況判定。

### 謹慎使用復活機制

「休息日」或「恢復日」可以成為有記憶點的衛教選擇，但建議只放在不計排行榜的練習模式，且不能暗示出現疼痛、疾病或停止條件後，休息一下就一定能安全繼續。

## 三、讓衛教內容真正被看見

最常見的浪費，是文案與對應函式都完成了，卻沒有接進實際遊戲流程。

### 建議的三層觸達設計

1. **即時回饋**：第一次遇到某事件或收集某道具時，多顯示一行簡短知識點。
2. **重複事件**：之後只顯示精簡數字或狀態，避免洗版。
3. **結算回顧**：依本局真正遇過的事件，整理三至五則重點與行動提醒。

不要只用搜尋程式碼確認文案「有被 import」。應建立測試，驗證每則內容確實有可達的觸發路徑。

### 衛教資料應獨立管理

建議每則內容至少包含：

```ts
interface EducationItem {
  id: string;
  topic: 'training' | 'injury' | 'nutrition' | 'safety';
  trigger: string;
  message: string;
  action: string;
  sourceLabel?: string;
  sourceUrl?: string;
  reviewedAt?: string;
}
```

將文案放在獨立資料檔，醫療或運動科學專業人員才能不必閱讀場景程式就完成審稿。

### 內容分層

- **一般衛教**：解釋概念，不針對個人下判斷。
- **行動提醒**：提供安全、非個人化的下一步。
- **紅旗警訊**：明確指出應停止活動或尋求緊急協助的情境。
- **免責聲明**：說明遊戲不取代醫療診斷、個別訓練或營養評估。

術語如 Zone 2、LSD 或間歇訓練，應說明「在本遊戲中代表什麼」，避免玩家把遊戲數值當成真實處方。

## 四、跳躍手感與障礙公平性

單鍵跑酷的美術可以簡單，但輸入必須可靠。

### Jump buffering

玩家在落地前稍早按下跳躍時，可暫存輸入並在落地瞬間起跳。

- **建議起始值**：約 100～150ms。
- 低幀率手機應先嘗試消耗有效輸入，再扣除該幀時間。
- 暫停、回首頁與重新開始時要清除緩衝，避免殘留輸入。

若玩家仍常覺得「已經落地卻不能跳」，可以評估加入短暫 coyote time，但需避免讓碰撞規則失去一致性。

### 障礙間距要同時考慮時間與距離

固定像素間距在高速時會縮短反應時間。可採用：

```text
最小生成距離 = max(固定安全像素, 當前速度 × 所需反應時間)
最小生成時間 = max(關卡最短時間, 最小生成距離 ÷ 當前速度)
```

「290px 在 440px/s 時約為 0.66 秒」只能作為公式示例；是否真的有問題，要看目前設定是否另有更長的時間下限，不能只看單一像素參數。

### 驗算跳躍幾何

基本公式：

```text
理論跳高 = 初速² ÷ (2 × 重力)
理論滯空時間 = 2 × 初速 ÷ 重力
水平跳距 ≈ 滯空時間 × 當前水平速度
```

應把這些數字和障礙高度、碰撞框、道具高度及最高速度一起驗證，確保所有允許生成的排列都有解。

### 其他手感細節

- 碰撞框要符合視覺預期；可略小，但「縮小 10～15%」只能當原型起始值。
- 受傷時同時提供角色狀態、文字與適度畫面回饋，不只改顏色。
- 無敵時間使用「剩餘毫秒」依遊戲 delta 累減，暫停時不得流失。
- 避免高頻閃爍與強烈震動；尊重 `prefers-reduced-motion`。
- 高速或低幀率時要測試穿透碰撞，必要時限制速度或採用更可靠的碰撞策略。

## 五、數值平衡：模擬是警報器，不是真人證據

體力、傷害、恢復、生成頻率與配速彼此耦合，不適合只靠開發者十幾次試玩調整。

### 建議方法

1. 將規則寫成純函式。
2. 建立固定種子或蒙地卡羅模擬。
3. 設計不同躲避率與收集率的玩家模型。
4. 輸出各關抵達率、完賽率、停止原因、平均分數與分數分布。
5. 將合理區間寫成回歸測試，避免後續改動意外破壞難度。

玩家模型參數、每組模擬次數與目標完賽率都是**專案假設**。例如「一般玩家完賽率 40～60%」可作為某些大眾型原型的起點，但不能當成所有衛教遊戲的業界標準。

模擬無法代表觸控延遲、學習速度、手機掉幀與真人策略。正式調整仍需要：

- 小規模真人試玩；
- 各關完成漏斗；
- 匿名、聚合的遊戲統計；
- 錯誤觀念與衛教理解訪談。

### 檢查分數天花板

如果所有完賽者的距離分相同，差異只剩少量固定道具加分，排行榜容易快速飽和。設計時要檢查：

- 高手之間是否有足夠差異；
- 是否出現大量同分；
- 分數是否鼓勵了不希望的行為；
- 差異化數值能否由伺服器可靠驗證。

## 六、手機網頁 UX、無障礙與效能

### 畫面與操作

- 手機直式可優先採 9:16，但仍要測試不同長寬比。
- 使用響應式縮放、置中與 safe-area，處理瀏海與瀏覽器工具列。
- 遊戲區域避免造成頁面上下捲動，但 DOM 按鈕保留原生焦點與鍵盤操作。
- 手機觸控按鈕建議至少約 44 CSS pixels。
- 空白鍵與方向鍵操作時，要排除焦點位於按鈕、連結或輸入框的情況。

### HUD 資訊層級

- 只保留當下決策需要的資訊。
- 體力與風險同時顯示文字、數字或圖示，不只靠顏色。
- 沒有特殊狀態時收合重複資訊。
- 即時訊息盡量控制在一至兩行，不遮住角色與障礙。
- HUD 使用 DOM 通常較容易處理中文、ARIA、觸控與響應式版面；若全用 canvas，也必須另行補足無障礙介面。

### 執行環境

- 所有系統依 delta time 更新，並對異常長 frame 做限制或分段。
- `visibilitychange` 時自動暫停，避免切換分頁或來電後產生巨大 delta。
- Canvas、WebGL 或 `AUTO` 應依實際效能、視覺需求與目標裝置測試決定，不能把 Canvas 當成通用最佳解。
- 測試低階 Android、Safari、120Hz 螢幕與低幀率情況。

### 音訊與素材

- Web Audio API 程式音樂可降低授權風險，但仍要遵守瀏覽器自動播放限制。
- 必須提供靜音開關，暫停時同步停止或暫停音樂。
- 外部圖片、字型與音效要確認授權；原創 placeholder 應留下清楚替換介面。

## 七、可維護架構：規則與畫面分離

建議的責任分工：

```text
src/
├── game/
│   ├── config/     # 遊戲平衡數值
│   ├── data/       # 衛教文案、標籤與來源
│   ├── systems/    # 純函式規則與狀態推進
│   ├── entities/   # 玩家、障礙、道具與視覺物件
│   ├── scenes/     # 整合物理、輸入與系統
│   └── events/     # 遊戲與 DOM UI 的事件介面
├── shared/         # 前後端共用型別、計分與驗證規則
├── services/       # API client
└── ui/             # 首頁、HUD、暫停、結算與分享
```

### 通用原則

- 所有平衡數值集中於設定檔，不散落在場景程式。
- 遊戲數值旁註明「這是遊戲平衡值，不代表真實生理數據」。
- 場景主要負責協調，不承擔所有計算。
- 前後端重算需要共用同一份規則或同一套版本定義。
- localStorage 資料要有版本與舊資料容錯。
- 一個長 frame 若跨過關卡或狀態到期，應分段套用正確倍率。
- 玩家狀態保留 `idle/running/jumping/hurt/gameOver/finished` 等介面，降低日後替換 sprite sheet 時需要改動的範圍；實際仍可能調整 preload、動畫時序與碰撞框。

## 八、計分、排行榜與分享

### 先定義分數要鼓勵什麼

分數公式應回答：

- 完成進度重要嗎？
- 收集道具重要嗎？
- 高風險選擇有沒有可理解的回報？
- 健康完成是否優先於硬撐？
- 同分時如何排名？

不要為了拉開分數而採用伺服器無法驗證的客戶端數值。

### 排行榜長期設計

- 完賽與中途停止分開或明確排序。
- 相同 outcome、分數與距離可採競賽名次，例如 `#1、#1、#3`。
- 規劃賽季、封存或排行榜重置。
- 避免每次排名都掃描整張歷史資料表。
- 監控索引、查詢成本、資料庫讀寫與免費額度。

### 分享功能

- Web Share API 不可用時，提供複製文字或儲存圖片的替代方案。
- 分享圖要在常見社群尺寸上保持安全邊界。
- 本專案採 1080×1080 方形分享卡；排行榜名次必須等伺服器提交成功後才能寫入圖片。
- 不在圖片或文字中暴露未經同意的真實姓名、健康資訊或裝置資料。

## 九、後端與防作弊的務實邊界

最重要的原則：

> **前端永遠不能直接決定排行榜分數。**

無帳號網頁遊戲可採用：

- 開局時由伺服器簽發一次性高熵 token；
- 資料庫只保存 token 雜湊並設定有效期；
- 週期性提交單調遞增的 elapsed time 與收集數；
- 結算由伺服器重算距離、分數與關卡；
- 檢查牆鐘時間、checkpoint、理論道具上限及狀態一致性；
- 對建立跑局與提交行為做頻率限制；
- 暱稱正規化、移除控制字元並限制長度；
- 動態文字一律用 `textContent`，避免 XSS。

### 隱私

- 不保存不必要的原始 IP、健康資訊或真實姓名。
- 若要依 IP 限流，可使用定期輪替金鑰的 HMAC 化識別值。
- 定義資料保存期限與清理策略。
- 公開排行榜提醒玩家不要輸入敏感資料。

### 誠實面對限制

瀏覽器程式可被修改。沒有帳號、遠端事件種子或可重播紀錄時，只能提高作弊成本，不能保證完全防作弊。有獎活動應另外評估帳號、Turnstile、固定種子及重播驗證。

checkpoint 週期、token 有效時間及完賽門檻都是實作參數。例如 10 秒 checkpoint 與 15 分鐘 TTL 只是本專案經驗，不是固定標準；應同時衡量防作弊、失敗恢復與雲端成本。

## 十、醫療內容、倫理與素材邊界

- 免責聲明要具體說明遊戲數值是虛構平衡值。
- 避免「你應該每週跑三次」等個人化處方式語言。
- 對疼痛、疾病與紅旗警訊採安全優先，不把「忍耐」設計成高分策略。
- 註明資料來源、內容版本與審閱日期，但不要暗示來源機構為遊戲背書。
- 不使用真實運動員肖像、知名角色、賽事識別或品牌 Logo，除非已取得授權。
- 角色、道具與音樂若是 placeholder，要列出並保留替換清單。

這些內容是產品與衛教設計原則，不構成法律意見；正式公開前仍應依地區、使用情境與蒐集資料種類評估法規需求。

## 十一、測試與驗收矩陣

### 1. 單元測試

- 體力、風險與恢復上下限；
- 狀態效果、保護、暫停與時間推進；
- 關卡、計分、生成安全及衛教選擇；
- localStorage 舊資料容錯；
- 伺服器驗證與排行榜排序。

### 2. 平衡模擬

- 各玩家模型的關卡抵達率；
- 完賽率與主要停止原因；
- 道具收集與分數分布；
- 必死排列與分數天花板；
- 固定種子回歸測試。

### 3. E2E

- 首頁、開始、canvas、跳躍；
- 暫停／繼續、聲音、重新開始；
- 結算、衛教、排行榜、分享；
- API 失敗與降級狀態；
- 桌機與手機 viewport。

開發測試鉤子必須由 `import.meta.env.DEV` 或等效機制保護，正式 bundle 不得暴露。

### 4. 真機與正式站

- 至少一支 iPhone 與一支中低階 Android；
- 320×568、主要手機尺寸與桌機；
- 中文可讀性、觸控命中與瀏覽器工具列；
- console/page errors、網路失敗與 API schema；
- 正式站 asset 版本、部署 commit SHA 與 production test hook 缺席。

每個需求最好能對應到「測試方式與證據」，不要只列完成勾選。

## 十二、部署、資料庫與營運

### 建議發布順序

1. 檢查 Git 變更與未追蹤檔案。
2. 執行 TypeScript、Lint、格式、單元測試與正式建置。
3. 建置 Functions／Workers。
4. 執行桌機與手機 E2E。
5. 在本機套用資料庫 migration 並測試。
6. 記錄目前已知正常的部署與 commit 作為 rollback point。
7. 依 schema 相容性決定順序：非破壞性 migration 可在相容程式部署前套用；breaking change 應採 expand → 部署相容程式 → backfill／migrate → contract，而不是固定「一律先 migration」。
8. 確認正式資料庫 ID、binding、備份／回滾策略與 migration 內容後，才執行 remote mutation。
9. Commit、push 或執行一次直接部署；避免兩套流程重複部署。
10. 確認正式部署為 Active，且 source SHA 等於推送的 commit。
11. 在正式網址檢查首頁、遊戲、console、唯讀 API 與主要流程。

### Cloudflare Pages／D1 本專案經驗

- Git push 不會自動替 D1 執行 migration。
- secret 不應寫入 repository；缺少時 API 應安全拒絕。
- 寫入 API 應驗證瀏覽器 `Origin` 與實際請求網址同源；preview 是否可寫入應用獨立環境開關與 D1 binding 管理，不要把部署權限混在單一固定 hostname 的檢查中。
- Wrangler log 在受限環境可導向專案內 `.wrangler/`。
- 靜態遊戲與排行榜 API 的額度要分開估算。
- 降低不必要的 checkpoint、全表排名與高頻資料庫查詢，通常比立刻升級方案更有效。

### 正式營運還要監控

- Workers 請求與錯誤率；
- D1 rows read／written、儲存與慢查詢；
- 各關停止率與衛教觸達率；
- 瀏覽器 console、API 錯誤與分享失敗；
- 內容更新日期、資料保留與舊賽季封存。

## 十三、建議開發順序

1. 定義學習目標、禁忌訊息與安全邊界。
2. 建立最小核心循環與灰盒角色。
3. 將規則、型別、config 與衛教資料分離。
4. 同步調整跳躍手感、生成規則與基礎數值。
5. 建立固定種子模擬與單元測試。
6. 將衛教接進首次事件與結算回顧。
7. 用真人與真機驗證玩法理解、手感與內容觸達。
8. 再加入排行榜、防作弊與分享。
9. 完成 E2E、migration、部署與 production smoke test。

不必在寫任何 Phaser 程式前就完成所有模擬；更務實的做法是讓**可測的規則層與灰盒原型並行迭代**，先證明核心循環可玩，再逐步投資美術與後端。

## 最重要的三句話

1. **好玩讓玩家留下，機制讓玩家形成理解，衛教內容讓玩家把正確觀念帶走。**
2. **模擬可以防止數值失控，但不能取代真人、真機與正式站驗證。**
3. **本機測試通過不等於部署完成；資料庫、部署版本與 production smoke test 缺一不可。**
